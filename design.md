# Otibeguni Design Reference

This file is the default visual reference for new mockups, UI explorations, and design-system work in this repo.

## Brand Colors

- `Otibeguni Blue`: `#01abf2`
- `Otibeguni Yellow`: `#ffe306`
- `Otibeguni Magenta`: `#fb2398`
- `Otibeguni Purple`: `#7811a7`

## Core Guidelines

- Design for light mode.
- Default page background should be white unless a specific component needs a different neutral surface.
- Do not use gradients.
- Prefer flat design over layered, glossy, glassy, or skeuomorphic treatments.
- Stick to the brand colors for accents, emphasis, and interactive states.
- Keep components visually consistent with the rest of the site rather than inventing a separate visual language for one-off mockups.
- Do not add eyebrow, kicker, or overline text by default. Use straightforward headings unless the existing page pattern already requires a small label.

## Practical UI Rules

- Prefer simple fills, borders, and spacing before adding shadows or decorative effects.
- If a secondary neutral is needed, use white, very light gray, or a subtle tint derived from the brand palette.
- Use rounded corners intentionally, but avoid overly soft or inflated component shapes unless the existing site already does that.
- Icons should feel optional and additive. Components should still read clearly if icons are removed.
- Avoid decorative section labels that repeat what the heading already says.
- Before handing off any UI work, check the page once for unintended extra spacing or crowding, especially around breadcrumbs, headings, section starts, and card grids.
- If a component places image and text together, check their alignment explicitly and make sure the text block aligns intentionally with the image rather than drifting vertically.
- Mockups should be implementation-oriented: they should resemble something that can plausibly ship in the current site.

## For Future System Work

- When proposing multiple UI options, vary structure and density first, not the entire visual language.
- Reuse this file as the baseline for future tokens, component states, and reusable patterns.
